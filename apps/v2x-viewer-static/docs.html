V2X API

What a detection looks like, how to upload one, and how to read it back.

Tech Stack (What We Use, and Why)

This system is a small, serverless pipeline for “many devices write small JSON events in real time”, with a read-only HTTP API for the viewer.

Here are the AWS services in the stack and what each one does: 

AWS IoT Core
  What it does:
    This is the device-facing "front door". Devices connect to AWS and send messages to named channels
    (topics). MQTT is a lightweight messaging protocol; TLS means the connection is encrypted.
  Why we use it:
    It is built for large numbers of devices and spotty networks. It supports per-device identities
    and permissions (each device can be limited to only publish to its own topic), without you running
    your own always-on server.

IoT Rule
  What it does:
    A server-side "router". When a message arrives on a topic pattern (for detections), this rule
    forwards the message to backend processing.
  Why we use it:
    It connects incoming device messages to your ingest code without a custom service in the middle.
    You can change routing behavior in configuration, not in device firmware.

AWS Lambda (Ingest)
  What it does:
    A small piece of server code that runs on demand. It receives one detection message, validates and
    normalizes fields (timestamps, numbers, required keys), and writes a record to DynamoDB.
  Why we use it:
    No servers to manage. It scales automatically with traffic and you pay only for execution time.

Amazon DynamoDB
  What it does:
    The database for detection items. Each detection is stored as a single JSON-like record (an item).
    DynamoDB supports fast lookups and time-ordered pagination when keys are modeled correctly.
  Why we use it:
    It handles very high write rates with low latency and scales without you sizing a database server.
    It also supports automatic expiration (TTL) so old detections can be cleaned up.

Amazon API Gateway
  What it does:
    The public HTTPS entrypoint for reads. It exposes URLs like `/detections/recent` and forwards the
    request to the read Lambda.
  Why we use it:
    It gives you a stable, managed URL, request routing, and browser-friendly behavior (CORS), without
    hosting a web server.

AWS Lambda (Read)
  What it does:
    Read-only server code. It queries DynamoDB (recent items, by object id, by geohash) and returns
    JSON to the browser, including pagination cursors when needed.
  Why we use it:
    It keeps database access off the client, lets you enforce a consistent response shape, and avoids
    exposing any internal-only fields.

AWS Amplify Hosting (Static)
  What it does:
    Hosts the viewer UI (plain HTML/CSS/JS) behind a CDN so it loads fast.
  Why we use it:
    It is the simplest way to deploy a static site on AWS. The browser only talks to the read API, so
    there are no AWS credentials in the UI.

Data flow diagram: 

Write path (device -> database)
Option A (simple HTTP upload)
Device
  -> API Gateway (POST /detections)
  -> Lambda (ingest)
  -> DynamoDB (stores detection item)

Option B (device messaging)
Device
  -> AWS IoT Core (message endpoint)
  -> IoT Rule (routes the message)
  -> Lambda (ingest)
  -> DynamoDB (stores detection item)

Read path (browser -> database)
Browser
  -> Amplify Hosting (serves viewer files)
  -> API Gateway (HTTPS)
  -> Lambda (read)
  -> DynamoDB (reads detection items)

JSON Structure

JSON shape for a single detection record (as returned by the read API).

{
  "event_id": "<string>",
  "object_id": "<string>",
  "object_type": "<string>",
  "timestamp_utc": "2026-02-05T20:59:19Z",
  "confidence_score": 0.0,
  "gps_location": {
    "latitude": 0.0,
    "longitude": 0.0
  },
  "geohash": "<string>",
  "street_name_normalized": "<string>",
  "global_context": {
    "city": "<string>",
    "state": "<string>",
    "country": "<string>"
  },
  "camera_data": {
    "image_reference_url": "<string>",
    "svo2_reference_url": "<string>",
    "bifocal_metadata": {}
  },
  "notes": "<string>",
  "device_id": "<string>",
  "ts_event": "2026-02-05T20:59:19Z#<event_id>",
  "expires_at": 0,
  "ingested_at_epoch": 0
}

Field Notes

event_id              Unique id for the event (UUID/ULID).
object_id             Stable identifier for the detected object.
object_type           Category label (e.g. traffic_cone).
timestamp_utc         ISO8601 timestamp in UTC (e.g. 2026-02-05T20:59:19Z).
confidence_score      0.0-1.0 confidence of detection/classification.
gps_location          latitude/longitude in decimal degrees (WGS84).
geohash               Geohash string used for coarse geo grouping / queries.
street_name_normalized
                      Optional normalized street name (snake_case).
global_context        City/state/country strings (optional but useful).
camera_data           References only; store large media in S3/KVS and link here.
notes                 Freeform note string.
device_id             Device identifier for the uploader.
ts_event              Sort key: timestamp_utc#event_id (used for ordering).
expires_at            TTL epoch seconds for DynamoDB item expiry.
ingested_at_epoch     Server ingest time (epoch seconds).

API Query Example

Fetch recent detections (paginated). The response includes items and an optional next cursor. 

curl 'https://qxacv7wah0.execute-api.us-west-1.amazonaws.com/detections/recent?limit=5'

{ "items": [ { "...": "..." } ], "next": "..." }

Device Upload Example

For now there is a public, unauthenticated HTTP endpoint that accepts a JSON detection payload. This is the simplest way to get data in. 

curl -X POST \
  -H 'content-type: application/json' \
  -d '{
    "event_id": "01J0TESTULID00000000000000",
    "device_id": "edge-device-001",
    "object_id": "traffic_cone_001",
    "object_type": "traffic_cone",
    "gps_location": { "latitude": 37.915425, "longitude": -122.33492 },
    "geohash": "f43h7",
    "timestamp_utc": "2026-02-05T20:59:19Z",
    "street_name_normalized": "owl_wy",
    "global_context": { "city": "San Francisco", "state": "CA", "country": "USA" },
    "confidence_score": 0.78,
    "camera_data": { "image_reference_url": "", "svo2_reference_url": "", "bifocal_metadata": {} },
    "notes": "Traffic cone example"
  }' \
  'https://qxacv7wah0.execute-api.us-west-1.amazonaws.com/detections'

Batch Upload Example

The same endpoint also accepts multiple detections in one request (either a JSON array, or a wrapper object with an items array). Batch uploads return per-item results and may partially succeed. Max: 500 items per request. 

curl -X POST \
  -H 'content-type: application/json' \
  -d '{
    "items": [
      { "object_id": "traffic_cone_001", "timestamp_utc": "2026-02-05T20:59:19Z" },
      { "object_id": "traffic_cone_002", "timestamp_utc": "2026-02-05T20:59:20Z" }
    ]
  }' \
  'https://qxacv7wah0.execute-api.us-west-1.amazonaws.com/detections'
